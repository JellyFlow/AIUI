# 分类

在 AIUI 框架中，如果从用户实际看到的交互形态来划分，智能体主要可以分为两类：`对话式 AIUI` 和 `沉浸式 AIUI`。二者的核心差别不在于“页面复杂度”，而在于 UI 是嵌在对话流里，还是由一个独立界面承载完整体验。

## 1. 对话式 AIUI

对话式 AIUI 指的是：UI 直接出现在聊天流中，以卡片、结果块、表单或工具面板的形式嵌在当前上下文里。用户一边和 AI 对话，一边在消息内部继续点击、确认、补充输入或查看结果。

这类 AIUI 的重点是：**让交互留在对话里完成**。用户不需要跳到另一个独立界面，也不需要中断当前会话，就可以完成查看、确认、筛选或轻量操作。

适合的场景包括：

- 天气卡片
- 搜索结果卡片
- 行程确认面板
- 待办确认和补充表单
- 快速查询与轻量任务反馈

它的典型特点包括：

- **交互承载**：界面附着在聊天流中，和消息上下文天然连续。
- **上下文特点**：通常共享当前会话上下文，适合承接一句问题之后的下一步操作。
- **开发重点**：把页面描述、参数结构和卡片交互设计好，让模型知道什么时候应该返回这类 UI。
- **体验目标**：快速反馈、低打断、一步到位。

## 2. 沉浸式 AIUI

沉浸式 AIUI 指的是：UI 不再附着在聊天消息里，而是由一个独立界面来承载完整的交互过程。用户的注意力主要停留在当前界面上，AI 则围绕这个界面的状态、操作和反馈持续参与协作。

这类 AIUI 的重点是：**让界面本身成为主要交互空间**。聊天、语音或其他输入方式仍然可以存在，但它们更多是在服务这个界面，而不是替代这个界面。

适合的场景包括：

- 斗地主或其他回合制小游戏
- 复杂工作流面板
- 设备控制界面
- 长时任务执行界面
- 需要持续状态反馈的工具或助手

它的典型特点包括：

- **交互承载**：界面是主舞台，聊天不再是唯一容器。
- **上下文特点**：通常以独立运行上下文维持状态和流程，更适合长时驻留。
- **开发重点**：需要自己组织界面状态、事件循环，以及必要时接入 `SSE`、`WebSocket`、`fetch` 等能力，构建完整交互链路。
- **体验目标**：持续协作、状态稳定、操作空间完整。

## 怎么选择？

- 如果你希望交互尽量轻、尽量快，并且留在当前聊天上下文里，优先选择 **对话式 AIUI**。
- 如果你希望界面本身成为主要操作空间，并围绕一个持续状态长期运行，优先选择 **沉浸式 AIUI**。
- 很多真实产品会同时使用这两类形态：用对话式 AIUI 承接轻量任务反馈，用沉浸式 AIUI 承接完整场景体验。
